Divide And Conquer React Components

Manage growing complexity by dividing components into separate files and folders.

Posted by Heikki Kupiainen/Oppikone on 15.07.2017 12:20:05

Avoid Big Files

In previous articles about creating React applications, I showed some examples on how to separate functionality and views into distinct components. In those examples, all components were placed inside the main source code file. However, when your app grows a little bigger than a simple Hello World, this solution will become increasingly unsatisfactory. To manage your code better, you will soon need to restructure it placing components inside their dedicated files. Moreover, you will need to regroup your software components into distinct folders in some logical way.

Therefore I created just for you a new branch showing how to divide your React components into more manageable fragments inside your file system!

Let's observe the solution in detail. You will see that App component's all subcomponents have been stripped off! They find their new home inside components folder. While this new folder structure might appear naively simplistic, let's not forget that our app is also quite naive at this point as well. No need to design a full blown fine tuned folder structure for our files when we don't really have anything yet to place inside those folders!

While removing subcomponents from your main App component, just add following lines to the beginning of the renewed slim App file:

import { AppTitle } from './component/AppTitle.js'
import { CarFilterList } from './component/CarFilterList.js'

And now that we import those components into main App, we also must make sure to export them as well so they can be imported in the first place! That is quite simple - just add export keyword before the declaration of a class:

export class AppTitle extends React.Component { ...

Can You Place Many Components in One File?

There is some debatte about placement of components and classes in separate files. Shall there be separate file for each component? Is it wise to place many components in one file? The answer generally depends on following factors:

  • Are components defined in file really needed elsewhere?
  • How many lines of code are in one file?

I think it is just okay to define react components that are not intended to be generic and reusable by many different parent components. It is just fine to define a React component that is only used by one specific parent component. If that is the case, there is definitely no need to create a separate file for that class. Neither it is necessary or even recommendable to declare such class exportable by adding export keyword before that file. One should not be too obsessed about creating "reusable" components. One important justification for creating non-reusable components is just to improve readibility of your main component - it can be achieved through structuring with help of "internal" components. Those components are not intended to be reused - they were formed only to serve structural clarity of their parent! Such components can be placed in one file with their parent component - which in turn then is exported. The file should be named after the main exportable component it contains.

The second question about file size is more tricky. I think that when a file exceeds 300 lines of code then there will be an increasing need to consider splitting the component into many files. But with modern editors that typically offer a structured view, it is actually possible to place all source code in one file only and it still remains perfectly navigable. There are even some benefits to this: coding is faster (at least in the beginning) when you don't need to create files and folders for your new components. But when you use a version management system and especially when there are many developers in the project - I would stick to upper limit of about 300 lines of code per file.
